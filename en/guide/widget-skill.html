Widget Skill ​

Widget Skill is the experimental UMG Widget Blueprint capability introduced in NexusFlow 0.2.0. It targets HUDs, panels, menus, status bars, and similar UI assets, allowing agents to create and modify Widget Blueprints through structured definitions.

This is currently an MVP / experimental feature. Start with small UIs, local changes, and tasks that can be verified through readback.

What It Can Do ​

  • Create Widget Blueprints from scratch.
  • Read existing Widget Blueprint structure and layout information.
  • Patch existing widget text, colors, percentages, slot data, event declarations, and binding declarations.
  • Rewrite the full Designer tree or a specific container branch.
  • Capture real preview screenshots when available.
  • Work with the Blueprint Skill to add events, variables, functions, and EventGraph logic.

Main Actions ​

ActionPurpose
generate_widget_blueprintCreate a complete Widget Blueprint
read_widget_blueprintRead structure, layout, and diagnostics
write_widget_treeReplace the full Designer tree or a container branch
patch_widget_treeModify property-level data on existing widgets
capture_widget_previewCapture a real preview screenshot

Recommended Workflow ​

text
1. Decide the UI type and viewport size.
2. Use the root Canvas for a few region containers such as Header, Content, Footer, or Modal.
3. Use Border, VerticalBox, HorizontalBox, ScrollBox, and SizeBox inside regions.
4. Give important widgets stable IDs; mark interactive widgets with variable.is_variable=true.
5. Run generate_widget_blueprint or write_widget_tree.
6. Use read_widget_blueprint to verify structure and layout diagnostics.
7. If events or bindings are needed, use the Blueprint Skill to add Graph logic.
8. Verify through compile, save, readback, or screenshots.

Tips ​

  • Widget Tree and Blueprint Graph are separate layers: Designer widgets belong in widget_tree; events, functions, variables, and graph logic belong in the general Blueprint definition.
  • Use the root Canvas for region containers; avoid scattering many TextBlocks or Buttons directly under it.
  • Use patch_widget_tree for small property-level edits; use write_widget_tree for hierarchy changes.
  • A successful write is not enough; verify the final UE state through read_widget_blueprint or screenshots.
  • For complex UIs, generate the static structure first, then add interaction logic incrementally.

Next Steps ​

  • Blueprint Operations: learn the general Blueprint Graph capabilities.
